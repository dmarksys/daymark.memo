poke: A Scheduled iMessage Nudge System

1. Problem

A habit prompt that arrives every day at 3:00 PM becomes wallpaper within a week. The brain learns the cadence and begins filtering out the content. A prompt that arrives at an unpredictable time—especially via a channel the user is already primed to take seriously—does not get filtered out as easily.

poke is a personal nudge system for macOS built on this premise. It sends short iMessages at randomized times inside a user-defined active window, with optional fixed-time messages for scheduled commitments and optional metronomic interval messages for steady cues. It runs as a per-user LaunchAgent and sends through imsg, a small CLI wrapper around the macOS Messages session.

The constraint keeps the system simple enough to audit in one sitting.

2. Runtime Model

poke is a short-lived CLI. launchd invokes poke tick every 300 seconds; each invocation runs for milliseconds and exits. This avoids the need for a sleep loop, fork, async runtime, or resident daemon.

The 5-minute cadence is set by StartInterval=300 in the LaunchAgent plist. RunAtLoad=true guarantees the first tick fires when the agent loads, so a fresh install does not wait up to five minutes before its first schedule generation.

Every tick reloads config, reloads state, and writes back before exit. Crash recovery is trivial—the next tick is a clean start. The only inputs are the config file, the state file, and the wall clock. Process supervision, log redirection, and restart on failure are all launchd's problem.

The tick interval StartInterval=300 is a default, not a structural commitment. The architecture is "short-lived CLI invoked by launchd"; the cadence at which launchd invokes it is a knob in the plist. The work per tick is trivial—read two files, possibly write one, check a timestamp—but waking 1,440 times a day instead of 288 fights harder with App Nap, timer coalescing, and idle sleep for no behavioral gain.

The cadence is also the floor for interval items: the every_minutes >= 5 validation in config.rs is a downstream consequence. A reminder every minute would recreate the noise the system exists to filter out.

A user who wants tighter resolution can lower both StartInterval in the plist and the every_minutes minimum in config.rs. Nothing else in the architecture is sensitive to the value.

A resident daemon would offer second-level precision in exchange for a process requiring supervision, restart logic, and careful handling across system sleep, user logout, and software updates. launchd handles all of that regardless of which cadence is chosen.

3. Message Types

Three kinds of pokes are merged into one pending queue per day. They share a delivery mechanism but differ in their invariants:

Random pokes are the primary mechanism. The user configures a pool of messages and a daily count (random_per_day); poke selects times and messages each morning. They obey minimum spacing and only fire inside [start_hour, end_hour).

Scheduled pokes are unconditional commitments at fixed wall-clock times—"send the afternoon check-in at 15:00." They fire regardless of the active window and do not interact with the random rotation. They exist because some prompts are not nudges but reminders, and the user wants exact control.

Interval pokes are metronomic. The user specifies an every_minutes; poke generates one slot at start_hour:00, then every every_minutes while before end_hour:00. Hydration and posture cues are the motivating cases. They are window-bound (no water reminders at 2 AM) but stand outside random rotation, so they do not perturb the variety logic.

4. Schedule Generation

Random poke times are sampled by segment, not by uniform draws over the full window.

Each day's random count is itself variable: a per-weekday baseline (or random_per_day.default) plus a symmetric uniform-integer jitter, clamped to at least one. This is selected once when the day's schedule is generated. Weekday baselines let a configuration encode "weekends are quieter"; the jitter keeps the daily count from being predictable even within a single weekday.

// src/schedule.rs L13-26 (92ea5aa3)

pub fn generate_for_date<R: Rng + ?Sized>(
    config: &Config,
    recent_history: &[RecentMessage],
    date: NaiveDate,
    rng: &mut R,
) -> anyhow::Result<Vec<PendingPoke>> {
    let interval = active_interval(date, config.schedule.start_hour, config.schedule.end_hour)?;
    let count = daily_random_count(&config.schedule.random_per_day, date, rng);
    let mut pending = generate_in_interval(config, recent_history, date, interval, count, rng)?;
    pending.extend(generate_scheduled_for_date(config, date)?);
    pending.extend(generate_interval_for_date(config, date, interval)?);
    pending.sort_by_key(|poke| poke.at);
    Ok(pending)
}

The active window is then divided into that many equal segments. Within each segment, one timestamp is sampled uniformly. The full sequence is then checked against random_min_spacing_minutes; if any adjacent pair is too close, the entire set is resampled. This loop runs up to 1,000 times before returning a density-infeasible error. Config validation rejects configurations whose worst-case count (max baseline + jitter) does not fit the window under the minimum spacing.

The obvious design was uniform sampling over the full window; with six pokes in a twelve-hour window it clusters visibly—three in the morning, none in the afternoon. Segment sampling makes even daily coverage a structural property rather than a probabilistic one.

Minimum spacing is enforced as a second pass rather than baked into the sampling, because the segment construction already handles most of the work—spacing violations only occur near segment boundaries, and rejection sampling there is cheap.

Interval pokes are generated deterministically: starting at start_hour:00, advance by every_minutes while strictly before end_hour:00. If the interval exceeds the window, exactly one poke is generated at start_hour:00. There is no randomness and no history dependency.

Scheduled pokes are generated one-per-configured-item at the configured wall-clock time. Friendly time formats ("3:00pm") are accepted in addition to "HH:MM".

All three sets are merged and sorted by timestamp before being written to pending.

5. Message Selection

Within the random pool, message selection treats categories as a first-class axis of rotation. The intent is that a user with three hydration messages, two movement messages, and one focus message does not see "drink water" three times in a row.

For each random slot in the day's count, selection proceeds as follows:

  1. If the number of remaining slots equals the number of remaining unseen messages, restrict candidate categories to those containing unseen messages. This guarantees that when the day's chosen count meets or exceeds the pool size, every configured message appears that day.
  2. If alternatives exist, exclude the category of the most recent message (preventing back-to-back categories).
  3. Among the surviving categories, prefer those still containing unseen messages.
  4. Within the chosen category, prefer unseen messages, and exclude the most recently used message if alternatives exist.
  5. Break ties by least-recently-used position in recent_history.

recent_history is a bounded list (8 entries) of the most recent successful sends, persisted across day boundaries. Without it, rotation resets at midnight—9:00 AM could show the same message as 8:30 PM the night before. The list keeps rotation continuous across the boundary.

History is only updated on successful sends of random pokes. Scheduled and interval pokes do not pollute the rotation: their selection is fixed by configuration, not by the algorithm, and folding them into history would make random rotation harder to reason about.

The alternative is a daily shuffle. It gives coverage trivially, but produces unsatisfying sequences—three movement pokes in a row, no continuity across midnight. LRU selection is more work for a result that feels varied rather than merely random.

6. Tick Behavior

A single tick performs the following:

  1. Acquire an exclusive file lock on state.lock. The lock is held for the entire tick.
  2. Load and validate config. Load state, or initialize empty state if no file exists.
  3. If last_schedule_date does not match today's local date, generate today's schedule, replace pending, clear sent, and persist.
  4. Find the earliest pending poke with at <= now that is eligible under the active window (random and interval pokes require the window; scheduled pokes do not).
  5. If none exists, exit zero.
  6. Send exactly one message via imsg.
  7. On success: move the poke from pending to sent, drop any other overdue pokes from pending, append to recent_history if the poke was random, and persist.
  8. On failure: log stdout, stderr, and exit status; preserve pending; exit nonzero.

Drop other overdue pokes on success. If the laptop was closed or the user was offline, several pokes may be overdue when the next tick runs. poke sends one—the earliest—and drops the rest. The alternative is a burst of stale prompts on wake, which is precisely the failure mode the system exists to avoid. Missed pokes are missed; there is no backfill.

Preserve the pending queue on failure. Almost every plausible failure mode for imsg is transient: the Messages session has not been logged in, the binary moved, Messages itself crashed. Preserving the queue lets the next tick retry without user intervention. If imsg fails persistently, every tick will fail loudly until the user investigates.

The single-message-per-tick rule is what makes both of these behaviors safe. If a tick sent everything overdue, the drop-on-success rule would be a data-loss bug rather than a feature.

7. State

State lives at $XDG_STATE_HOME/poke/state.json, falling back to ~/.local/state/poke/state.json if XDG_STATE_HOME is unset, empty, or relative. The schema:

{
  "last_schedule_date": "2026-04-19",
  "pending": [ /* PendingPoke */ ],
  "sent": [ /* SentPoke */ ],
  "recent_history": [ /* RecentMessage */ ]
}

Atomic writes. State is written via the temp-file-and-rename pattern: serialize to .state.json.tmp, fsync, rename to state.json, then best-effort fsync the parent directory. The rename is atomic on POSIX filesystems, so a crash mid-write leaves the previous state intact.

File locking. state.lock is acquired with fs2::FileExt::lock_exclusive at the start of every tick and regen, and held until the function exits. launchd's StartInterval does not guarantee non-overlapping invocations; if a tick runs longer than 300 seconds (it should not, but it could), the lock prevents a second tick from sending a duplicate message.

Human-readable JSON. State is an operator artifact. A reasonable response to "did my last poke fire?" is cat state.json | jq .sent. Binary formats—bincode, CBOR—would be marginally faster and would offer no advantage at the scale of one file per user containing a few dozen entries. Their cost is opacity. JSON wins for the same reason the rest of the system was designed the way it was: auditability.

State migrations are handled by serde defaults. The kind field on pokes was added after the initial release; legacy state without that field deserializes with kind = "random". The same pattern applies to category and recent_history. This keeps the state format forward-compatible without requiring migration code.

8. Delivery

imsg is invoked directly as a subprocess:

imsg send --to DESTINATION --text MESSAGE

Arguments are passed as a slice to std::process::Command, not a shell string. Message text containing apostrophes, dollar signs, or backticks cannot break out of its argument; there is no quoting to get wrong. The function ImsgSender::send is the only place in the codebase that spawns a subprocess, and it is small enough to read at a glance.

imsg_path must be absolute and is validated at config load time for runtime commands. launchd runs poke in a minimal environment where PATH is unreliable; relying on PATH resolution would produce intermittent failures that are easy to misdiagnose.

The subprocess captures exit status, stdout, and stderr. A nonzero exit triggers the failure path described in §6: stdout and stderr are logged via the LaunchAgent's StandardOutPath and StandardErrorPath, the pending queue is preserved, and poke exits nonzero so that launchd's failure tracking sees the error.

There is no delivery confirmation. A zero exit from imsg indicates that the Messages session accepted the message, not that the recipient received it.

9. Known Limitations

  • Tick cadence is a tuning default, not a structural constraint. The plist sets StartInterval=300; interval items shorter than 5 minutes are rejected at config load as a downstream effect. Both values can be lowered in tandem if a user wants tighter resolution. The default exists for resource frugality and as a guard against metronomic burnout (see §2).
  • Single message per tick. By design. A burst is worse than a miss.
  • No delivery confirmation. Inherited from imsg and the macOS Messages session.
  • DST transitions. chrono::LocalResult::Ambiguous is resolved to the first interpretation. Pokes scheduled during the ambiguous hour may fire at the wrong wall-clock time on the transition day; the error is bounded by one hour and not retried.
  • No config hot-reload. Config is read fresh on every tick, but there is no watch mechanism. Changes take effect on the next tick—within 5 minutes—which is sufficient for an interactive operator.
  • No multi-device routing. poke has no knowledge of which device will receive the message, nor any fallback channel if iMessage is unavailable.
  • Missed pokes are gone. No catch-up. This is the price of the burst-suppression rule in §6---acceptable given the primary intended purpose of the tool.

10. Summary

Most of the interesting code for poke is in the schedule generation and message selection logic. The rest is plumbing. The goal throughout was to minimize surface area and delegate everything else to launchd. An operator can audit the full system by reading three files and running poke show.
