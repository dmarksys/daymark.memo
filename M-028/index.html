Understanding HNSW Graph Search

Context

Hierarchical Navigable Small World (HNSW) graphs represent one of the most efficient approaches to approximate nearest neighbor search in high-dimensional spaces. The algorithm constructs a multi-layer graph structure that enables logarithmic search complexity while maintaining high recall.

This memo examines the core mechanisms of HNSW, its theoretical foundations, and practical performance characteristics.

Problem

Exact nearest neighbor search in high-dimensional spaces suffers from the curse of dimensionality. Traditional tree-based structures like KD-trees degrade to linear search as dimensionality increases. For vector databases serving semantic search or recommendation systems, we need sub-linear query time with acceptable recall.

The challenge is constructing a graph that:

  • Enables efficient greedy search from any entry point
  • Maintains connectivity across distance scales
  • Supports incremental updates without full reconstruction

Mechanism

HNSW builds on the concept of navigable small world networks, extended to a hierarchical structure.

Layer Construction

The algorithm assigns each node to layers probabilistically:

$ \ell = \lfloor -\ln(\text{uniform}(0,1)) \cdot m_L \rfloor $

where $m_L$ is a normalization factor controlling layer scaling. This creates an exponentially decaying layer distribution similar to skip lists.

Insertion Process

When inserting a new vector $q$:

  1. Enter at the top layer
  2. Greedily traverse to the nearest node at current layer
  3. Descend to next layer and search neighborhood
  4. At layer 0, connect to $M$ nearest neighbors
  5. For layers $\ell > 0$, connect to $M_{\text{max}}$ neighbors

The greedy search at each layer uses the distance metric to navigate the graph:

def search_layer(q, entry_points, layer, ef):
    visited = set()
    candidates = []
    for ep in entry_points:
        dist = distance(q, ep)
        heappush(candidates, (dist, ep))
        visited.add(ep)

    while candidates:
        current_dist, current = heappop(candidates)

        for neighbor in neighbors(current, layer):

Search Complexity

The hierarchical structure provides $O(\log n)$ expected hops through layers, with each hop examining $O(1)$ neighbors when properly tuned. Query time becomes:

$ T_{\text{query}} = O(m \cdot \log n) $

where $m$ is the connectivity parameter.

Analysis

The key advantage is maintaining high recall while achieving logarithmic search complexity.

Parameter Sensitivity

The construction parameters critically impact performance:

  • M: Number of connections per node. Higher $M$ improves recall but increases memory and construction time. Typical values: 16-64.
  • efConstruction: Size of dynamic candidate list during construction. Controls build quality. Typical values: 100-400.
  • efSearch: Size of dynamic candidate list during search. Controls recall-latency tradeoff. Adjusted per query requirements.

Limitations

HNSW has several constraints:

  1. Memory overhead: Each node stores multiple links across layers, typically 2-4x the raw vector size
  2. Update costs: Deletions require tombstoning or graph repair
  3. Dimension sensitivity: Performance degrades in dimensions >1000 without dimensionality reduction
  4. Non-metric spaces: Requires triangle inequality for correctness guarantees

Implications

HNSW has become the default choice for production vector databases including Pinecone, Weaviate, and Qdrant. The algorithm's combination of strong theoretical guarantees and practical performance makes it suitable for:

  • Semantic search over document embeddings
  • Image similarity retrieval
  • Recommendation systems with learned embeddings
  • Real-time anomaly detection

For applications requiring exact results or supporting complex filtering predicates, hybrid approaches combining HNSW with inverted indexes or metadata filtering are necessary.

The algorithm's efficiency enables serving billion-scale vector collections with single-digit millisecond latencies, fundamentally changing what's architecturally feasible for semantic search systems.
